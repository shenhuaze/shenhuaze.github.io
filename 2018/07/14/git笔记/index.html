Git笔记

Posted on 2018-07-14 | In Git | | Visitors:

工作中会用到git，这里参考廖雪峰的git教程，记录一些常用的git操作。

创建本地仓库

新建一个文件，并将它添加到本地仓库中

新建文件或者修改文件之后，如果需要commit，都要先git add filename，然后再git commit -m "message"。如果需要写多行commit信息，有两种方法，一种是用单引号替代双引号，先敲一个单引号，即git commit -m '，然后换行写commit信息，写完之后再敲第二个单引号，然后回车；另一种方法是不加-m参数，直接敲git commit，然后会弹出来一个vim编辑器，在最上方可以写多行commit信息。

查看当前git状态

任何时候，都可以用git status查看当前的状态。

查看文件哪些地方被修改了

如果一个文件被修改了，我们可以用git diff filename查看文件有哪些地方被修改了。

如果我们commit了多个版本，可以用git log查看commit历史。

如果忘了版本号，可以用git reflog查看。

撤销修改

如果我们修改了一个文件，但是还没有将它添加到暂存区中。可以用git checkout -- filename来撤销对该文件的修改。

如果我们修改了一个文件，并且已经将它添加到暂存区中了，则先用git reset HEAD filename，撤销将它添加到暂存区的操作，然后再用git checkout -- filename，撤销对文件内容的修改。

如果我们修改了一个文件，并且已经添加到暂存区和commit了，那就用git reset --hard HEAD^回退到上一个版本。

如果已经将修改提交到远程库了，就只有手动修改回去了。

删除文件

如果现在需要从版本库中删除一个文件，则先手动将这个文件删除，然后用git rm filename，将它从版本库中去掉。

如果我们手动误删了一个文件，可以用git checkout -- filename从版本库中恢复这个文件，意思就是撤销对该文件的修改。

远程仓库

如果需要和别人合作，就需要将代码上传到远程仓库，比如公司的代码库或者github都是远程仓库。使用远程仓库一般需要在远程仓库的设置里面添加本机的公钥。

分支管理策略

在合并分支的时候，可以加上--no-ff参数，强制禁用fast forward模式，保留分支信息。

实际开发中，应按照几个基本原则进行分支管理：

  1. master分支应该是非常稳定的，仅用来发布新版本，平时不能在上面干活；
  2. 平时主要都在dev分支上干活，到某个时候，比如发布1.0版本的时候，再把dev分支合并到master分支上，在master分支上发布1.0版本；
  3. 团队的每个人都在dev分支上干活，每个人都有自己的分支，时不时往dev分支上合并就可以了。

bug分支

如果你正在dev上干活，正干到一半，master分支上出了一个bug，需要尽快解决。我们可以先用git stash命令，将dev分支上的工作状态先保存起来，然后切换到master分支上，然后新建一个分支，比如issue-101，再issue-101分支上把bug解决掉之后，切换回master分支，将issue-101分支合并到master分支上，然后删掉issue-101分支。

然后切换回dev分支，用git stash list查看之前保存的工作状态历史。

然后可以用git stash apply stash@{0}恢复之前的工作状态。

还可以用git stash drop stash@{0}删除之前保存的工作状态。

强制删除未合并的分支

如果我们想删除一个还未合并的分支，比如feature分支，如果用之前的命令git branch -d feature来删除它，会报错。

为了强制删除未合并的分支，可以将参数-d改为-D，就可以删除了。

Rebase

git rebase可以将本地未push的分叉提交历史整理成直线，目的是使得我们在查看历史提交的变化时更容易，因为分叉的提交需要三方对比。

使用github

通过github，可以让别人参与自己的开源项目，自己也可以参与别人的开源项目。

如果要参与别人的开源项目，假设项目名叫demo，可以在别人的开源项目右上角点击Fork，然后在我们的github账户下就会克隆出一份demo项目。然后就可以将自己账户下的demo项目克隆到本地。

如果我们想修复demo项目的一个bug，或者添加一个新功能，立刻就可以开始干活，干完之后，就往自己的github上推送。如果我们希望demo项目的拥有者接受我们的修改，就可以在github上发起一个pull request，然后对方就会看到你的请求，决定是否接受你的修改。
